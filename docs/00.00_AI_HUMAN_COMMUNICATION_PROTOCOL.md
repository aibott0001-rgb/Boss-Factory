1. 🌌 EXECUTIVE SUMMARY: THE "ONE-STEP" DOCTRINE
This document defines the only acceptable mode of interaction between the CEO and any AI Agent within the Boss Factory ecosystem. It eliminates confusion, prevents catastrophic errors, and ensures that every action is verified before the next begins.
The Core Philosophy:
"Speed comes from precision, not haste. One perfect step is faster than ten broken ones."
Any AI agent violating this protocol by providing multi-step instructions, assuming success without verification, or skipping context ingestion is considered non-compliant and must be halted immediately.
2. 🧬 THE GOLDEN LOOP: STANDARD OPERATING PROCEDURE
Every single task, from fixing a typo to deploying a new module, must follow this strict 4-Phase Loop. No exceptions.
Phase 1: Context Ingestion (The "Boot")
Before generating any output, the AI MUST:
Read 00.00 (This File): Acknowledge the communication rules.
Read 00.5_SYSTEM_EXECUTION_PROTOCOL.md: Understand the technical boot sequence.
Scan 11_PROGRESS_STATE_KERNEL.md: Identify the exact current state and the single next task.
Check 17_ERROR_FORENSICS_LOG.md: Verify no known blockers exist for the intended task.
Phase 2: Single-Action Proposal (The "Ask")
The AI MUST:
Analyze the task.
Break it down into the smallest possible atomic unit (e.g., "Create one file," "Run one command").
Present ONLY ONE clear command or code block to the CEO.
STOP and wait. Do not generate further steps. Do not explain future steps.
Explicitly state: "Please execute this command and paste the full output. I will wait."
Phase 3: Verification & Execution (The "Handshake")
The CEO (Human) Will:
Execute the single command.
Copy the entire terminal output (success or error).
Paste it back to the AI.
The AI MUST:
Analyze the output.
If Success: Confirm verbally, update internal state, and proceed to Phase 4 (Commit) OR propose the next single action (return to Phase 2).
If Error: Analyze the specific error message. Check 17_ERROR_FORENSICS_LOG.md. Propose a single fix command. Return to Phase 2.
NEVER assume the command worked without seeing the output.
Phase 4: The Guardian Gate (The "Seal")
Upon completion of a logical unit (e.g., a feature is built):
The AI MUST instruct the CEO to run: ./scripts/guardian.sh.
The AI MUST wait for the Guardian's success confirmation.
Only then is the task marked DONE in 11_PROGRESS_STATE_KERNEL.md.
3. 📜 RULES OF ENGAGEMENT
Rule #1: The "One-Action" Mandate
Violation: Providing a list of 5 commands to run sequentially.
Correction: Provide Command #1. Wait for output. Then provide Command #2.
Reasoning: If Command #2 fails because #1 had a hidden warning, the system breaks. Sequential verification prevents cascade failures.
Rule #2: Explicit Confirmation for Destructive Actions
Definition: Any command involving rm, git reset, drop, delete, or overwriting files.
Protocol: The AI must ask: "⚠️ WARNING: This command will [Effect]. Are you ready to execute? (Yes/No)"
Execution: Proceed ONLY after a clear "Yes".
Rule #3: Output Transparency & Forensics
Requirement: The CEO must paste the full, unedited terminal output.
AI Duty: If an error occurs, the AI must:
Identify the root cause.
Check if it exists in 17_ERROR_FORENSICS_LOG.md.
If New: Log it to the Forensics Log (via Guardian or manual entry) before attempting a fix.
If Known: Apply the stored fix immediately.
Rule #4: Context Preservation & Handoff
Scenario: The CEO returns after a break or switches AI models.
Protocol: The AI must immediately request: "Let's verify our state. Please run cat docs/11_PROGRESS_STATE_KERNEL.md and paste the result."
Never guess the current status. Always read the source of truth.
Rule #5: The "Stop" Condition
The CEO can type "STOP" or "PAUSE" at any time.
The AI must immediately halt all generation, summarize the current partial step, and wait for further instructions.

5. 🚨 EMERGENCY PROTOCOLS
Scenario A: The "Loop of Death" (Repeated Failures)
If the same error occurs 3 times:
AI Action: Halt immediately. Do not suggest a 4th variation.
Statement: "We are in an error loop. I am escalating this to the CEO for manual review. Please check docs/17_ERROR_FORENSICS_LOG.md and provide additional context."
Resolution: CEO intervenes, provides new info, AI resets context.
Scenario B: Lost Context / Confusion
If the AI seems to hallucinate or forget previous steps:
CEO Command: Type "RESET CONTEXT".
AI Action: Immediately stop. Request the CEO to run git status and cat docs/11_PROGRESS_STATE_KERNEL.md. Re-boot from Phase 1.
Scenario C: Corrupted Local State
If files are missing or git is broken:
AI Command: Guide the CEO to run: git fetch origin && git reset --hard origin/main && git clean -fd.
Verification: Run ./scripts/guardian.sh to confirm health.
