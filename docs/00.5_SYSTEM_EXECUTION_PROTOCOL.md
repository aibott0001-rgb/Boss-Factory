## 2. THE BOOT SEQUENCE (Read Before Coding)
Before writing a single line of code, every Agent MUST execute these 4 steps:

### Phase A: Context Ingestion (5 Minutes)
1.  **Read `docs/00_MASTER_BLUEPRINT.md`**: Understand the vision, zero-cost mandate, and omni-present architecture.
2.  **Read `docs/00.5_SYSTEM_EXECUTION_PROTOCOL.md`** (This File): Understand the rules of engagement.
3.  **Read `docs/11_PROGRESS_STATE_KERNEL.md`**: Identify the **exact** current progress and the **Next Immediate Action**.
4.  **Scan `docs/17_ERROR_FORENSICS_LOG.md`**: Check for known errors and their fixes. **DO NOT repeat past mistakes.**

### Phase B: Environment Validation (2 Minutes)
1.  Run `./scripts/guardian.sh` immediately upon starting.
    *   **If Pass:** Proceed to coding.
    *   **If Fail:** Read the error, check `17_ERROR_FORENSICS_LOG.md` for a fix, apply it, and re-run. Do not code new features until the build is green.
2.  Verify `.env.local` exists (but never commit it).

### Phase C: Task Execution (The Work)
1.  Select the task from `11_PROGRESS_STATE_KERNEL.md`.
2.  Write code following the **Golden Rules** (Section 4).
3.  Test locally (`npm run dev`).

### Phase D: The Guardian Gate (Completion)
1.  Run `./scripts/guardian.sh`.
2.  **If Success:** The script commits, pushes, and updates progress. Task is **DONE**.
3.  **If Failure:** The script logs the error to `17_ERROR_FORENSICS_LOG.md`. Fix the error and re-run Guardian.

## 3. HUMAN-AI COMMUNICATION PROTOCOL
To prevent confusion and ensure high-quality output, all interactions between the CEO (Human) and AI Agents must follow this strict format:

### Rule 1: One Action At A Time
- **AI Mandate:** Never provide multiple complex steps in one response. Provide **ONE** clear command or code block.
- **Human Mandate:** Execute the single command, then paste the **full terminal output** back to the AI.
- **Stop Condition:** The AI must **WAIT** for the user's confirmation ("Done", "Success", or output log) before proceeding to the next step.

### Rule 2: Explicit Confirmation
- Before executing any destructive command (e.g., `rm`, `git reset`), the AI must ask: *"Are you ready to execute [Command]? This will [Effect]."*
- The AI must not proceed without a clear "Yes".

### Rule 3: Output Transparency
- If a command fails, the AI must analyze the **exact error message** provided by the user.
- The AI must never assume success. It must verify via user feedback or a verification command (e.g., `ls`, `git status`).

### Rule 4: Context Preservation
- At the start of every new session, the AI must read `docs/11_PROGRESS_STATE_KERNEL.md` and `docs/17_ERROR_FORENSICS_LOG.md` to regain context.
- If the user says "I lost context" or "Let's restart", the AI must immediately guide the user to run `git reset --hard origin/main` and re-read the Kernel.
